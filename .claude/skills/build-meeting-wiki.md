# Build Meeting Wiki Index

Generate `Knowledge/meetings/WIKI-INDEX.md` -- a reverse index that maps entities (people, projects, tools, partnerships) to the meetings where they appear. This turns meeting files into a queryable knowledge graph without creating hundreds of separate entity pages.

## Why This Exists

Meeting notes are the richest source of context in PersonalOS, but they're organized chronologically. When you need to answer "what have we discussed about X?" or "which meetings involved Y?", you'd need to grep through everything. The wiki index inverts this: look up the entity, see every meeting where it came up.

## Output Format

`Knowledge/meetings/WIKI-INDEX.md`:

```markdown
# Meeting Wiki Index

> Auto-generated by `/build-meeting-wiki` on YYYY-MM-DD HH:MM
> Covers: YYYY-MM to YYYY-MM (X meetings across Y months + Z recurring series)

## People

### Internal (your team/org)
| Person | Role/Context | Meetings | Last Seen |
|--------|-------------|----------|-----------|
| Alice | Engineering lead | [2026-04-07](...), [2026-04-01](...) | 2026-04-07 |

### External
| Person | Company | Meetings | Last Seen |
|--------|---------|----------|-----------|
| Bob | Partner Co | [2026-03-XX](...) | 2026-03-XX |

## Projects & Initiatives

| Project | Status | Meetings | First Mentioned | Last Mentioned |
|---------|--------|----------|-----------------|----------------|
| Feature Launch | In progress | 15 meetings | 2025-12-XX | 2026-04-01 |

## Tools & Technologies

| Tool/Tech | Context | Meetings |
|-----------|---------|----------|
| Kubernetes | Infrastructure migration | 5 meetings |

## Partnerships

| Partner | Status | Meetings |
|---------|--------|----------|
| Vendor X | Active integration | 4 meetings |

## Recurring Themes

| Theme | Description | Meetings |
|-------|-------------|----------|
| Compliance | Legal review process | 8 meetings |

## Cross-Reference: Meetings to Entities

Reverse lookup -- for each meeting, what entities were discussed:

### 2026-04
| Meeting | People | Projects | Tools |
|---------|--------|----------|-------|
| 2026-04-07 Team sync | Alice, Bob | Feature Launch | k8s |

### 2026-03
...

---
*Last rebuilt: YYYY-MM-DD HH:MM*
*Next suggested rebuild: YYYY-MM-DD (14 days)*
```

## Steps

### Step 1: Determine scope

- If `$ARGUMENTS` is "incremental" or empty: read the existing WIKI-INDEX.md footer for the last build date. Only process meetings created/modified since then.
- If `$ARGUMENTS` is "full": process all meetings from scratch.
- If no WIKI-INDEX.md exists, do a full build.

### Step 2: Collect meeting files

1. Glob for all `.md` files in `Knowledge/meetings/YYYY-MM/` directories
2. For incremental: filter to only files modified since last build date
3. Also read all `Knowledge/meetings/recurring/*/META.md` files (always, since these track active state)

### Step 3: Seed entity recognition

Before scanning meetings, read these files to build a list of known entities:
- `GOALS.md` -- projects, people, tools mentioned in goals
- `Knowledge/decision-log.md` (if exists) -- project names, people, dates
- Recurring meeting META.md files -- active topics, participants

This prevents missing known entities during extraction.

### Step 4: Extract entities from each meeting

For each meeting file, read it and extract:

- **People:** Names mentioned as participants or in the body. Classify as internal or external based on context.
- **Projects:** References to known initiatives and any new project names.
- **Tools/Technologies:** Product names, services, platforms.
- **Partnerships:** Named company partnerships or vendor references.
- **Themes:** Recurring topics (legal, compliance, hiring, training, architecture, etc.)

Use context from:
- The meeting's participant list and summary
- The meeting body text
- Known entities from the seed step

### Step 5: Merge with existing index

For incremental builds:
1. Read the existing WIKI-INDEX.md
2. Add new meeting references to existing entities
3. Add any newly discovered entities
4. Update "Last Seen" / "Last Mentioned" dates
5. Update meeting counts

### Step 6: Build cross-reference table

For the last 3 months of meetings, create the reverse lookup table (meeting -> entities). Older months get summarized as counts only to keep the file manageable.

### Step 7: Write WIKI-INDEX.md

- Write the complete index to `Knowledge/meetings/WIKI-INDEX.md`
- Include the build timestamp and suggested next rebuild date (14 days)
- Keep total file under 500 lines. If exceeding, trim the cross-reference table to last 2 months.

### Step 8: Report

Show the user:
- Meetings processed (new vs total)
- Entities found (people, projects, tools, partnerships, themes)
- Any new entities discovered that weren't in the previous index
- Suggested actions (e.g., "3 meetings mention 'NewCo' -- should this be tracked as a partnership?")

## Performance Rules

- Read meetings in batches of 10-15 using parallel Read calls
- Don't read the full content of meetings older than 6 months (use filename + first 20 lines only)
- For entity extraction, prefer scanning summary sections and action items over full transcript text
- If a meeting has a clear "## Summary" section, read only that section for entity extraction
- Seed entity recognition from GOALS.md and recurring META.md files before scanning meetings
